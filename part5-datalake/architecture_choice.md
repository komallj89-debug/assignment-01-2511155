## Architecture Recommendation

For a fast-growing food delivery startup that collects GPS location logs, customer text reviews, payment transactions, and restaurant menu images, I would recommend a **Data Lakehouse** architecture.

A Data Lakehouse combines the flexibility of a Data Lake (ability to store any data format) with the query performance and governance of a Data Warehouse (structured tables, ACID transactions, SQL analytics). Here are three specific reasons why this is the right choice:

**1. The startup collects fundamentally heterogeneous data.** GPS logs are high-volume time-series data in formats like JSON or Parquet. Customer text reviews are unstructured strings. Payment transactions are highly structured relational records requiring ACID guarantees. Restaurant menu images are binary blobs. A traditional Data Warehouse cannot store images or unstructured text. A pure Data Lake can store everything but makes it very difficult to run reliable SQL analytics on the structured payment data. A Data Lakehouse (e.g., built on Delta Lake, Apache Iceberg, or Databricks) handles all these formats within a single architecture — storing raw files while providing table-format layers for reliable querying.

**2. The startup is fast-growing and data requirements will change rapidly.** A Data Warehouse requires upfront schema definition, which is expensive to change as the business evolves (adding new delivery partner integrations, new review categories, etc.). A Data Lake accepts schema-on-read, meaning new data types can be ingested immediately without redesigning the schema. The Lakehouse gives the best of both: flexibility to ingest new data immediately, with the option to define schemas and quality constraints on mature, well-understood datasets like payments.

**3. The startup needs both operational analytics and AI/ML capabilities from the same platform.** Payment transactions need low-latency dashboards (OLAP queries). Customer reviews and GPS data need ML pipelines for sentiment analysis and route optimization. A Lakehouse supports both use cases natively — SQL analytics engines (like Spark SQL or DuckDB) can run directly on the lake files, while the same data is accessible to ML frameworks like TensorFlow or PyTorch. Maintaining a separate Data Warehouse and Data Lake would mean duplicating data and managing two separate pipelines, which adds cost and synchronization complexity.
